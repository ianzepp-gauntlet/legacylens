## Themes

### 1. Benchmark-Driven Engineering

Rather than picking defaults by intuition, every configuration decision was backed by measured data. The pipeline went: design benchmark harness → run 16-config sweep → pick winner → commit change → run expanded 40-query and 209-query suites → confirm → cache results. This produced a documented, reproducible evidence trail for why `llama-text-embed-v2` at 1024 dimensions with paragraph chunking became the production default.

### 2. Layered Caching Architecture

The caching strategy evolved through at least four distinct iterations within the window:
- Day 2: In-memory L1 (LLM answers) and L2 (search results) with env flags
- Day 3 evening: L1 JSON cache removed; SSE source event parsing hardened
- Day 3 evening: L2 removed from UI and server entirely (over-engineering detected, reverted)
- Day 4: L1 reinstated as a pre-sharded on-disk cache with per-request UI toggle and OpenRouter cache_control hints for prompt caching

The willingness to add, remove, and re-add cache layers based on observed behaviour — rather than defending initial design choices — is a healthy engineering signal.

### 3. Streaming Transport Migration

SSE (Server-Sent Events) was the first streaming mechanism (Day 3, midday). By Day 3 evening it was replaced with WebSocket JSONL, which allowed source references to be streamed as discrete events before the LLM generation completed — a meaningful UX improvement. The migration touched the server (`web/app.py`), the client (`index.html`), and the test suite (`test_websocket_stream.py`) in a single coherent refactor, and a proper done/ack shutdown handshake was added the next morning to harden the protocol edge cases.

### 4. Test Coverage as a First-Class Output

Testing was present from the first commit (`3a67e91` on Day 2). By the end of the period, the test suite comprised at least 9 test files:
- `test_benchmark.py`, `test_chain.py`, `test_chunker.py`, `test_ingest.py`, `test_parser.py`
- `test_refactor_units.py`, `test_testability_refactor.py`, `test_vectorstore_refactor.py`
- `test_web_api.py`, `test_web_cache_key.py`, `test_websocket_stream.py`

The testability refactor commit (`76e19a5`) is particularly notable because it improved production code and added new test coverage simultaneously — an unusually clean coupling.

### 5. Aesthetic Coherence

The Fallout Pip-Boy / COBOL mainframe terminal theme was locked in on Day 2 and remained consistent. Font choices (Monofonto, IBM 3270), the greenbar paper effect, sprocket holes, and green-screen chunk viewer all reinforce a unified visual metaphor. This is not cosmetic trivia — for a COBOL explorer, the aesthetic signals domain competence to the target audience.

### 6. Multi-Model LLM Flexibility

The system was extended to support OpenAI GPT-4.1/5.x, OpenRouter (as a gateway to non-OpenAI models), and Gemini 2.5 Flash Lite. The default was switched to Gemini 2.5 Flash Lite via OpenRouter on Day 3 — a cost/latency optimisation move documented by the benchmark data. OpenRouter's `cache_control` hint was added on Day 4 to exploit prompt caching for the system message, further reducing per-request cost.

## Observations

**1. This is a submission-deadline sprint, not routine development.**
The README gained a "final submission requirements status checklist" on Day 4. The MVP requirements checklist and live demo URL were added on Day 3. The pace, the breadth of features shipped, and the explicit tracking of submission criteria all point to a course project or hackathon submission in its final 72 hours.

**2. The benchmark data volume is misleading.**
165,335 gross insertions sounds like a massive codebase expansion. It is not. Once the four benchmark result JSON/CSV dump commits are excluded, the actual source code growth is 8,357 lines across 3 days — significant but not outsized for a greenfield RAG application. The benchmark artefacts should ideally have been excluded from version control via `.gitignore` (which was partially remedied by `a1c6a39` on Day 4).

**3. Caching strategy shows productive experimentation.**
The cache system went through at least four distinct design states in three days. This is not instability — it is evidence of a developer who measures (via the benchmark suite) and responds to data rather than defending initial designs. The final state (on-disk sharded L1 with UI toggle, OpenRouter prompt caching, per-request L1 Trim) is substantially more sophisticated than what was present on Day 2.

**4. The testability refactor (`76e19a5`) is the most technically significant commit.**
It touched all six core pipeline modules, added 531 lines of new tests in three new test files, and left the code measurably more modular. Doing this mid-sprint — rather than after the deadline — indicates the developer understood that test infrastructure would pay back immediately in debugging time, not just after the project ends.

**5. Production readiness was treated as a Day 1 concern.**
Railway deployment config (`Procfile`, `runtime.txt`, Python version pin) appeared on Day 2. Production bugs were fixed as discovered rather than deferred. The LLM warmup on first page load, OpenRouter cache control hints, and sharded pre-computed caches are all latency-reduction features that only matter in production — not in local development. This orientation distinguishes the work from a "demo-only" project.

**6. The March 3 intensity suggests flow state, not thrashing.**
46 commits in one day with coherent messages, multiple large refactors that land cleanly, and progressive feature layering (SSE → WebSocket, no-cache → L1/L2 → L1-only → on-disk L1) all suggest the developer was in a highly productive flow state. The commit cadence in the 14:00–15:00 hour (16 commits) is the peak — likely a combination of pre-existing work being committed in rapid sequence as features came together.

**7. Minor concern: commit granularity on Day 3 midday.**
Several consecutive commits (e.g., `11f9f1c` → `b2308e7` → `9101ebe`) represent minor UI adjustments that could have been batched. This is not a significant issue, but it suggests the commit flow was somewhat stream-of-consciousness during rapid iteration. The later commits on Day 3 and all of Day 4 show better grouping of related changes.
